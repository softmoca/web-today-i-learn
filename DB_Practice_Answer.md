2. **MySQL에서 사용되는 `AUTO_INCREMENT`는 왜 필요할까?**
    - 정의 - Insert 시 자동으로 1씩 증가하는 값 생성
    - 매번 ID를 직접 넣으면
      → 중복 위험 
      → 빠뜨릴 위험 
      → 여러 요청이 동시에 들어올 때 충돌
    - AUTO_INCREMENT를 쓰면
      → INSERT 시 자동으로 1씩 증가
      → 항상 **유일한 값 보장**
      → 동시 요청에도 안전
    - 즉, AUTO_INCREMENT는 개발자의 실수를 줄이고, 동시성 상황에서도 안전하게 유일한 식별자를 생성하기 위해 필요하다.

        - NULL은 값이 아니라 ‘없음’

        - 프론트 처리
            - NULL 그대로 보여주면 UX 안 좋음
            - 보통, “미퇴군”,”- “, “기록없음”

5. **출석 시스템에서 동시에 100명이 등교 버튼을 누른다면 어떤 일이 일어날까? 이 문제를 [2026 공통강의 - DB](https://techcourse.woowahan.com/s/uLX8Ma07/ls/62meRAXd)에서 배운 트랜잭션과 ACID 속성으로 설명해보자.**
    - 한 사용자가 중복 요청을 보내거나 네트워크 지연으로 동일한 요청이 여러 번 처리되면 출석 데이터가 `중복으로 저장`(**Isolation (격리성)** 문제)될 수 있고, 처리 도중 오류가 발생하면 일부만 저장(**Atomicity (원자성)** 문제)되어 `출석이 누락되는 문제`도 발생할 수 있다.
    - 또한 여러 요청이 동시에 현재 출석 인원을 조회한 뒤 데이터를 저장하는 과정에서 서로의 작업을 인지하지 못하면 `정원 제한을 초과하는 등의 격리성 깨짐이 발생`할 수 있다.

   이러한 문제는 트랜잭션의 ACID 속성을 통해 해결할 수 있다.

    - `원자성`(Atomicity)은 출석 처리 과정을 하나의 작업으로 묶어, 중간에 실패하면 전체를 롤백하여 데이터가 부분적으로 저장되는 것을 방지한다.
    - `격리성`(Isolation)은 동시에 실행되는 트랜잭션들이 서로 간섭하지 않도록 보장하여, 동시 요청 상황에서도 정원 초과나 중복 저장과 같은 문제를 막는다.
    - 또한 `일관성`(Consistency)은 하루에 한 번만 출석할 수 있도록 하는 제약 조건을 유지하게 해주며, (트랜잭션이 끝나면 DB는 항상 규칙을 만족하는 정상 상태다)
        - DB에는 반드시 지켜야 하는 규칙이 있다
            - crew_id는 반드시 존재해야 함 (FK)
            - nickname은 중복되면 안됨 (UNIQUE)
            - 하루 1번 출석 (비즈니스 룰)
        - Consistency가 깨진 상태 예시
            - 존재하지 않는 crew_id로 출석 있음
            - 같은 사람이 하루에 2번 출석
            - nickname이 중복됨
    - `지속성`(Durability)은 한 번 저장된 출석 데이터가 시스템 장애가 발생하더라도 안전하게 보존되도록 한다.
    - 결국 트랜잭션과 ACID 속성은 다수의 사용자가 동시에 요청하는 상황에서도 출석 데이터의 정확성과 신뢰성을 보장하는 핵심 요소이다.
    - ✔️ 문제 상황 (트랜잭션 없을 때) - 100명이 동시에 INSERT / UPDATE 수행
        - 같은 crew가 중복 기록 생성
        - start_time 덮어쓰기 충돌
        - 데이터 꼬임 (race condition)
    - ✔️ ACID로 해결
        - 1. Atomicity (원자성) - “출석 기록은 전부 성공 or 전부 실패”
            - INSERT 도중 오류 발생 → 롤백
            - 중간 상태 없음
        - 2. Consistency (일관성) - “DB 규칙 항상 유지”
            - crew_id FK 깨지면 실패
            - 중복 출석 제한 조건 유지
        - 3. Isolation (격리성) ⭐ - “동시에 눌러도 서로 간섭 없음”
            - DB가 락 / MVCC로 충돌 방지
            - 한 요청이 끝나기 전 다른 요청이 영향을 못 줌
            - 동시에 INSERT → 내부적으로 순차 처리처럼 보이게 함
        - 4. Durability (지속성) - “저장된 출석은 절대 사라지지 않음”
            - 서버 죽어도 기록 유지
6. **출석 데이터가 파일(CSV)이 아닌 데이터베이스에 저장되는 이유는 무엇일까? 파일 시스템으로 출석을 관리했다면 어떤 문제가 생길까?**
    - 실제 운영 환경에서 요구되는 데이터의 **일관성, 동시성 처리, 확장성**을 파일 시스템만으로는 안정적으로 보장하기 어렵기 때문이다.
    - 만약 출석을 CSV 파일로 관리한다면 여러 문제가 발생할 수 있다.
        - 먼저, `데이터 중복 문제`가 생긴다. 사용자별 출석 정보를 여러 파일로 나누어 관리하거나, 같은 파일에 여러 번 기록되는 경우 중복 데이터가 쉽게 발생하며 이를 제어하기 어렵다.
        - 또한 `일관성 문제가 발`생한다. 예를 들어 사용자가 출석을 취소하거나 수정해야 할 때, 여러 위치의 데이터를 동시에 수정해야 하는데 일부만 반영되면 데이터가 서로 다른 상태로 남게 된다.
        - `동시 접근 문제`도 매우 크다. 여러 사용자가 동시에 파일에 접근해 출석을 기록하려고 하면, 파일이 깨지거나 일부 데이터가 덮어쓰기 되는 등 레이스 컨디션이 발생할 수 있다. 파일 시스템은 기본적으로 트랜잭션이나 락을 통한 정교한 동시성 제어를 제공하지 않기 때문에 이러한 문제를 안정적으로 해결하기 어렵다.
        - 또한 `보안 측면에서도 취약`하다. CSV 파일은 접근 권한 관리가 제한적이고, 누가 어떤 데이터를 수정했는지 추적하기 어렵기 때문에 악의적인 수정이나 데이터 변조를 방지하기 힘들다.
        - 반면 데이터베이스는 트랜잭션을 통해 원자성과 격리성을 보장하고, 제약 조건을 통해 데이터 일관성을 유지하며, 동시에 많은 사용자의 요청을 안정적으로 처리할 수 있다. 따라서 출석 시스템처럼 다수의 사용자가 동시에 접근하고 데이터의 정확성이 중요한 경우에는 파일이 아닌 데이터베이스를 사용하는 것이 필수적이다.
7. **출석 데이터를 관계형 DB가 아닌 NoSQL(예: MongoDB)로 저장한다면 테이블 구조가 어떻게 달라질까? 어떤 장단점이 있을까?**
    - 출석 데이터를 관계형 DB가 아닌 NoSQL(예: MongoDB)에 저장한다면, 가장 큰 변화는 **테이블 중심 구조에서 문서 중심 구조로 바뀐다는 점**이다.
    - 관계형 DB에서는 보통 `users`, `attendance` 같은 테이블을 분리하고, `user_id`를 기준으로 관계를 맺는다. 예를 들어 출석 테이블에는 `user_id`, `date`, `status` 같은 컬럼이 존재하고, 필요한 경우 JOIN을 통해 사용자 정보를 조회한다.
    - 반면 MongoDB와 같은 NoSQL에서는 하나의 문서(document)에 데이터를 함께 저장하는 방식이 일반적이다. 예를 들어 다음과 같이 사용자 단위로 출석 데이터를 포함할 수 있다.

    ```java
    {
      "userId": 1,
      "name": "영철",
      "attendance": [
        { "date": "2026-03-30", "status": "present" },
        { "date": "2026-03-29", "status": "late" }
      ]
    }
    ```

- 즉, 관계형 DB처럼 데이터를 나누고 JOIN하는 대신, **조회에 필요한 데이터를 한 번에 가져올 수 있도록 중첩 구조로 설계**하는 것이 특징이다. 이러한 구조 변화는 장단점을 만든다.

---

- 먼저 장점은 **`조회 성능과 유연성`**이다. 출석 정보를 조회할 때 별도의 JOIN 없이 한 번의 쿼리로 사용자와 출석 데이터를 함께 가져올 수 있기 때문에 읽기 성능이 좋다. 또한 스키마가 `유연`하여, 특정 사용자에게만 “지각 사유”, “체온 기록” 같은 필드를 추가하는 것도 쉽게 가능하다. 이런 점은 크루 프로필처럼 사용자마다 다른 정보를 가지는 경우 특히 유리하다.
- 반면 단점도 존재한다. 가장 큰 문제는 **`데이터 중복과 일관성 관리`**이다. 사용자 정보가 여러 문서에 중복 저장되거나, 출석 데이터를 여러 방식으로 저장하면 수정 시 모든 데이터를 함께 변경해야 하므로 일관성이 깨질 위험이 있다. 또한 관계형 DB처럼 강력한 제약 조건이나 JOIN 기반의 복잡한 조회가 어렵기 때문에, 출석 집계(예: 전체 출석률 계산)나 정`합성이 중요한 경우에는 설계가 더 까다로`워진다.

---

- 정리하면, 출석 데이터처럼 구조가 명확하고 정합성이 중요한 경우에는 관계형 DB가 더 적합하고, 크루 프로필처럼 구조가 자주 바뀌고 유연성이 필요한 데이터는 NoSQL이 더 적합하다.

1. **왜 crew 테이블에서 nickname을 기본키로 하지 않은 걸까? attendance 테이블에 attendance_id가 존재하는 이유는 무엇일까?**
    - 왜 nickname(자연키)을 기본키로 안 했을까?
        - 문제점
            1. 변경 가능성 (가장 치명적)
                - 닉네임 변경 → PK 변경 필요 → 모든 참조 테이블 수정 필요 (FK cascade)
            2. 중복 가능성  - 같은닉네임을 허용해야하는 경우
            3. 길고 비효육적
    - 왜 attendance_id가 필요한가? - “각 출석 기록을 유일하게 식별하기 위해”
        - 출석은 사실 (crew_id, attendance_date) 복합키로도 구분가능 하지만
        1. 유연성 부족 - 하루 2번 출석 → PK 깨짐
            1. 비즈니스 요구 변경 = DB 구조 깨짐
        2. PK가 길어짐 - join,fk 둘다 복잡해짐
        3. 수정 어려움 - 날짜 수정 → PK 변경 ⇒PK는 절대 안바뀌는 값이어야함
            1. 위험한 작업 - 참조하는 모든 FK 영향,CASCADE 필요
            2. 데이터 무결성 깨질 위험
            3. 성능 비용 큼
        - 그래서 attendance_id (PK) - 단일, 불변, 단순
    - nickname은 변경 가능하고 중복 가능성이 있기 때문에 기본키로 부적합하며, crew_id 같은 대리키를 사용하는 것이 안정성과 성능 측면에서 더 적절하다.
    - 또한 attendance 테이블에서는 각 출석 기록을 단순하고 유일하게 식별하기 위해 복합키 대신 attendance_id라는 대리키를 사용하는 것이 유지보수와 확장성에 유리하다.
2. **데이터베이스 제약 조건 중 RESTRICT, CASCADE는 무엇인가?**
    - RESTRICT와 CASCADE는 외래키(Foreign Key) 제약 조건에서 **참조 무결성을 유지하기 위한 동작 방식**을 정의하는 옵션이다. 즉, 부모 테이블의 데이터가 변경되거나 삭제될 때, 이를 참조하고 있는 자식 테이블의 데이터를 어떻게 처리할지를 결정한다.
    - 먼저 **RESTRICT**는 부모 데이터를 삭제하거나 수정하려 할 때, 해당 데이터를 참조하고 있는 자식 데이터가 하나라도 존재하면 그 작업을 **아예 막는 방식**이다. 예를 들어 사용자가 작성한 게시글이 남아 있는 상태에서 사용자 계정을 삭제하려고 하면, 게시글이 존재하기 때문에 삭제가 실패한다. 이는 데이터의 연결 관계를 강하게 유지하여, 잘못된 삭제로 인해 데이터가 깨지는 것을 방지하는 데 목적이 있다.
    - 반면 **CASCADE**는 부모 데이터가 삭제되거나 수정될 때, 이를 참조하는 자식 데이터도 **함께 자동으로 삭제(또는 수정)되는 방식**이다. 예를 들어 사용자가 탈퇴하면 그 사용자가 작성한 게시글, 댓글 등이 모두 함께 삭제되도록 설정할 수 있다. 이는 연관된 데이터를 일관되게 정리할 수 있다는 장점이 있다.
    - 실제 서비스에서는 상황에 따라 선택이 달라진다. 데이터 보존이 중요한 경우(예: 주문 내역, 결제 기록)는 RESTRICT를 사용해 삭제를 막거나, 논리 삭제(soft delete)를 선택하는 경우가 많다. 반대로 사용자와 강하게 종속된 데이터(예: 임시 데이터, 개인 설정, 좋아요 등)는 CASCADE를 사용해 함께 삭제하는 것이 관리 측면에서 더 적합하다.
    - 정리하면, RESTRICT는 “참조 중이면 삭제 금지”, CASCADE는 “부모 삭제 시 자식도 함께 삭제”라는 전략이며, 서비스의 데이터 정책에 따라 적절히 선택해야 한다.
3. **다음 두 쿼리는 동일한 결과를 반환하지만 성능에 차이가 있다. 어떤 차이가 있으며, 어떤 상황에서 각각 유리할까?**

    ```sql
    -- 쿼리 1: 서브쿼리 사용
    SELECT * FROM attendance WHERE crew_id IN (SELECT crew_id FROM crew WHERE nickname LIKE '네%');
    
    -- 쿼리 2: JOIN 사용
    SELECT a.* FROM attendance a JOIN crew c ON a.crew_id = c.crew_id WHERE c.nickname LIKE '네%';
    ```

    - 두 쿼리는 결과적으로 `nickname`이 `네`로 시작하는 크루의 출석 데이터를 조회한다는 점에서는 같지만, **실행 방식과 최적화 여지에서 차이**가 있다.
    - 먼저 첫 번째 쿼리는 `crew` 테이블에서 조건에 맞는 `crew_id` 목록을 먼저 구한 뒤, 그 값을 이용해 `attendance`를 조회하는 방식이다. 논리적으로는 “먼저 대상 크루를 찾고, 그다음 출석을 찾는다”는 흐름이라 이해하기 쉽다. 특히 조회 조건이 비교적 단순하고, 먼저 필터링된 ID 집합을 기준으로 검색하는 상황에서는 직관적으로 작성하기 좋다. 다만 DBMS와 실행 계획에 따라서는 서브쿼리 결과를 별도로 처리해야 하거나, 최적화가 충분히 되지 않으면 `비효율이` 생길 수 있다.
    - 반면 두 번째 쿼리는 `attendance`와 `crew`를 직접 조인한 뒤 `nickname` 조건으로 필터링하는 방식이다. 일반적으로 실무에서는 이 방식이 더 많이 쓰이며, DB 옵티마이저가 조인 순서나 인덱스 활용을 더 적극적으로 최적화할 수 있는 경우가 많다. 특히 `attendance.crew_id`, `crew.crew_id`에 인덱스가 있고, `crew.nickname` 검색 조건도 적절히 활용된다면 JOIN 방식이 더 안정적으로 좋은 성능을 낼 가능성이 크다. 또한 이후에 출석 데이터뿐 아니라 크루 이름, 상태 등 다른 테이블의 컬럼까지 함께 조회해야 하는 경우에도 JOIN이 더 자연스럽고 확장성이 좋다.
    - 정리하면, **서브쿼리는 읽기 쉽고 “대상 집합을 먼저 구한다”는 사고 흐름에 잘 맞을 때 유리**하고, **JOIN은 실무에서 더 일반적이며 대용량 데이터나 확장된 조회, 최적화 측면에서 더 유리한 경우가 많다**고 볼 수 있다. 다만 실제 성능은 단순히 문법 차이로만 결정되는 것이 아니라, 데이터 양, 인덱스 유무, DBMS의 옵티마이저 성능, 실행 계획에 따라 달라지므로, 실무에서는 결국 `EXPLAIN`으로 실행 계획을 확인하는 것이 가장 중요하다.
4. **attendance 테이블을 완전히 정규화하면 어떤 장점이 있을까? 반대로 일부 비정규화를 적용한다면 어떤 쿼리 성능 이점을 얻을 수 있을까?**
    - attendance 테이블을 완전히 정규화하면 `데이터 중복이 제거`되어 저장 `공간이 효율적으로 사용`되고, 하나의 데이터를 수정했을 때 관련된 모든 곳에 일관되게 반영되므로 `데이터 무결성과 일관성을 유지`하기 쉬워진다. 또한 crew와 attendance처럼 역할이 분리되기 때문에 책임이 명확해지고, 잘못된 데이터 삽입이나 참조 오류를 외래키와 같은 제약 조건으로 방지할 수 있어 전체 `시스템의 안정성이 높`아진다.
    - 반대로 일부 비정규화를 적용하면 `조회 성능 측면에서 이점`을 얻을 수 있다. 예를 들어 attendance 테이블에 nickname을 함께 저장해두면 매번 crew 테이블과 JOIN하지 않아도 되어 쿼리가 단순해지고 실행 속도가 빨라진다. 특히 대량 데이터에서 빈번한 조회가 발생하는 경우 JOIN 비용을 줄일 수 있어 성능이 개선된다. 다만 이 경우 데이터가 중복 저장되기 때문에 nickname 변경 시 여러 데이터를 함께 수정해야 하는 문제가 발생할 수 있어, 정규화와 성능 사이에서 상황에 맞는 균형을 잡는 것이 중요하다.
5. **출석 시스템이 수백 명의 사용자에 의해 동시에 접근된다면, 연결 풀링(connection pooling)은 무엇이고 왜 필요한가?**
    - 출석 시스템에 수백 명의 사용자가 동시에 접근하면, `각 요청마다 데이터베이스에 연결을 새로 생성하고 종료하는 과정이 반복`되는데 이 과정은 생각보다 비용이 크고 느리다. 데이터베이스 연결은 단순한 함수 호출이 아니라 네트워크 연결 생성, 인증, 세션 초기화 등의 과정을 포함하기 때문에, 요청마다 이를 반복하면 응답 속도가 느려지고 DB 서버에도 큰 부담이 된다.
    - 이때 사용하는 것이 **커넥션 풀링(connection pooling)**이다. 커넥션 풀링은 미리 일정 개수의 데이터베이스 연결을 생성해 풀(pool)에 보관해두고, 요청이 들어올 때마다 `새로운 연결을 만드는 대신 기존 연결을 빌려 쓰고 반납`하는 방식이다. 즉, “필요할 때마다 새로 만드는 것”이 아니라 “미리 만들어 둔 것을 재사용”하는 구조다.
    - 이 방식의 가장 큰 장점은 성능 향상이다. 매 요청마다 연결을 생성하지 않기 때문에 응답 속도가 빨라지고, DB에 가해지는 부하도 줄어든다. 또한 동시에 많은 요청이 들어와도 풀의 크기만큼만 연결을 유지하므로 DB 자원을 안정적으로 관리할 수 있다. 만약 커넥션 풀이 없다면 수백 명의 사용자가 동시에 접근할 때 수백 개의 연결이 순간적으로 생성되며, 이는 DB 서버의 커넥션 한계를 초과하거나 전체 시스템 성능 저하로 이어질 수 있다.
    - 따라서 커넥션 풀링은 단순한 성능 최적화가 아니라, 다수의 사용자가 동시에 접근하는 환경에서 안정적인 서비스 운영을 위해 필수적인 구조라고 볼 수 있다.
6. **실습에서 수행한 INSERT, UPDATE, DELETE를 하나의 트랜잭션으로 묶는다면 어떻게 작성할 수 있을까? 만약 DELETE 도중 오류가 발생하면 앞서 수행한 INSERT와 UPDATE는 어떻게 되어야 할까?**
    - 실습에서 수행한 `INSERT`, `UPDATE`, `DELETE`를 하나의 트랜잭션으로 묶으려면, 세 작업을 각각 독립적으로 실행하는 것이 아니라 하나의 작업 단위로 실행되도록 `BEGIN`(또는 `START TRANSACTION`)으로 시작하고, 모든 작업이 정상적으로 끝났을 때 `COMMIT`, 중간에 문제가 생기면 `ROLLBACK` 하는 형태로 작성하면 된다.
    - 즉, “셋 중 하나라도 실패하면 전부 취소”되도록 만드는 것이다. 이는 트랜잭션의 원자성(Atomicity) 개념과 연결된다.

    ```sql
    START TRANSACTION;
    
    INSERT INTO attendance (crew_id, attendance_date, start_time, end_time)
    VALUES (1, '2025-03-20', '09:00', '18:00');
    
    UPDATE attendance
    SET start_time = '09:05'
    WHERE attendance_id = 3;
    
    DELETE FROM attendance
    WHERE attendance_id = 7;
    
    COMMIT;
    ```

    - 하지만 만약 `DELETE`를 수행하는 도중 오류가 발생한다면, 앞에서 성공했던 `INSERT`와 `UPDATE`도 그대로 남아 있으면 안 된다. 왜냐하면 이 세 작업은 하나의 논리적인 작업 단위로 묶였기 때문이다. 따라서 이 경우에는 `ROLLBACK`을 수행해서 트랜잭션 시작 전 상태로 모두 되돌려야 한다. 즉, 새로 추가한 출석 기록도 사라지고, 수정한 출석 시간도 원래 값으로 복구되어야 한다. 이것이 바로 “all or nothing”이다.
